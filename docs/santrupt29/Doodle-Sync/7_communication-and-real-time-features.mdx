# Communication and Real-time Features

This section details how Doodle-Sync leverages real-time communication technologies to ensure a seamless and interactive user experience, particularly for chat, drawing updates, and game state synchronization.

## WebSocket Communication

WebSockets are employed to facilitate bi-directional, real-time communication between the server and connected clients. This allows for instant delivery of messages, such as chat messages, drawing strokes, and game events, without the overhead of traditional HTTP polling.

The `SimpMessagingTemplate` from Spring's STOMP support is a core component, enabling the broadcasting of messages to specific topics.

In `ChatMessageHandler`, guesses are received via STOMP at `/app/room.{roomCode}.guess` and broadcasted to `/topic/room.{roomCode}.chat` for all players to see.

```java
// chat-service/src/main/java/com/pm/chatservice/handler/ChatMessageHandler.java
@Controller
@Slf4j
@RequiredArgsConstructor
public class ChatMessageHandler {
    // ... other dependencies
    private final SimpMessagingTemplate broker;

    @MessageMapping("/room.{roomCode}.guess")
    public void handleGuess(
            @DestinationVariable String roomCode,
            @Payload GuessMessage guessMessage
            ) {
        String chatTopic = "/topic/room."+roomCode+".chat";
        // ... guess validation and broadcasting logic
        broadcast(chatTopic, "CHAT",
                guessMessage.getUsername(), guessMessage.getGuess(),
                guessMessage.getPlayerId());
        // ...
    }

    private void broadcast(String topic, String type,
                           String username, String message,
                           String playerId) {
        broker.convertAndSend(topic, ChatMessage.builder()
                .type(type)
                .username(username)
                .message(message)
                .playerId(playerId)
                .timestamp(System.currentTimeMillis())
                .build());
    }
    // ...
}
```

Similarly, in the `drawing-service`, `StrokeBroadcastListener` subscribes to Redis Pub/Sub channels for drawing events and uses `SimpMessagingTemplate` to broadcast these strokes to clients connected to the canvas topic `/topic/room.{roomCode}.canvas`.

```java
// drawing-service/src/main/java/com/pm/drawingservice/pubsub/StrokeBroadcastListener.java
@Component
@Slf4j
@RequiredArgsConstructor
public class StrokeBroadcastListener implements MessageListener {

    private final SimpMessagingTemplate broker;
    private final ObjectMapper objectMapper;

    @Override
    public void onMessage(Message message, byte[] pattern) {
        try {
            String channel = new String(message.getChannel());
            String roomCode = channel.substring("draw:".length());
            String json = new String(message.getBody());

            StrokeEvent stroke = objectMapper.readValue(json, StrokeEvent.class);
            String topic = "/topic/room." + roomCode + ".canvas";
            broker.convertAndSend(topic, stroke);
            // ...
        } catch (Exception e) {
            log.error("Failed to broadcast stroke from Redis Pub/Sub: {}",
                    e.getMessage());
        }
    }
}
```

## Message Queuing with Kafka

Kafka is utilized for more durable and asynchronous event handling, particularly for game-related events that might require persistence or more complex processing chains. While WebSockets provide real-time immediacy, Kafka ensures that critical events are not lost, even in the face of transient failures.

The `GameEventConsumer` in the `game-service` listens to the `game-events` Kafka topic. Currently, it's configured to log incoming hints, but it serves as an example of how Kafka can be used to decouple event producers from consumers.

```java
// game-service/src/main/java/com/pm/gameservice/kafka/GameEventConsumer.java
@Component
@RequiredArgsConstructor
@Slf4j
public class GameEventConsumer {
    // ...
    @KafkaListener(topics = "game-events", groupId = "game-service-hints")
    public void consume(ConsumerRecord<String, String> record) {
        String value = record.value();
        if (value.startsWith("HINT:")) {
            // re-publish to a dedicated hints topic for WebSocket
            // (Week 3 — for now just log it)
            log.info("Hint received: {}", value);
        }
    }
}
```

## Resilience and Error Handling

To ensure reliability, particularly for operations involving external systems like Redis and Kafka, the application employs retry mechanisms. The `@Retry` annotation from Resilience4j is used to automatically retry failed operations, such as writing to Redis when marking a player's correct guess. This prevents data loss due to transient network issues or service outages.

```java
// chat-service/src/main/java/com/pm/chatservice/handler/ChatMessageHandler.java
@Controller
@Slf4j
@RequiredArgsConstructor
public class ChatMessageHandler {
    // ...
    @Retry(name = "kafka-consumer") // Note: Annotation name implies Kafka consumer, but used for Redis write here. Best practice might be to rename or use a dedicated config.
    private void markPlayerGuessed(String key) {
        redisTemplate.opsForValue().set(key, "1",
                java.time.Duration.ofMinutes(5));
    }
    // ...
}
```

## Key Takeaways

*   **Real-time Interaction:** WebSockets via STOMP provide immediate feedback for chat messages and drawing updates.
*   **Event Durability:** Kafka ensures that critical game events are processed reliably, even with transient failures.
*   **Fault Tolerance:** Resilience4j's retry mechanism enhances the robustness of operations involving external data stores and message brokers.

## Integration Details

The system orchestrates communication by integrating WebSocket messaging for immediate user-facing updates with Kafka for backend event processing and persistence. Redis serves as a low-latency store for game state, such as correct guesses and words to be guessed, which are accessed by both the chat and drawing services.
