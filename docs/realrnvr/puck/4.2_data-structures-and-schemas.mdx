# Data Structures and Schemas

This section details the data formats and validation schemas employed throughout the application, ensuring data integrity and consistency across the client and server.

## User Input Schemas (Zod)

The client-side utilizes Zod for robust schema validation of user input, particularly for sign-up processes. This ensures that data conforms to predefined rules before being sent to the server.

### `SignUpSchema`

The `SignUpSchema` defines the expected structure and constraints for user registration data. It validates `username`, `email`, and `password` fields.

```typescript
import { z } from "zod";

export const SignUpSchema = z.object({
  username: z
    .string()
    .min(4, "At least 4 characters")
    .max(50, "Cannot exceed 50 characters")
    .trim()
    .regex(/^[a-zA-Z0-9._]+$/, {
      message: "Only letters, numbers, dots, and underscores allowed.",
    })
    .refine((val) => !val.startsWith(".") && !val.startsWith("_"), {
      message: "Cannot start with a dot or underscore",
    })
    .refine((val) => !val.endsWith(".") && !val.endsWith("_"), {
      message: "Cannot end with a dot or underscore",
    }),
  email: z.string().email("Valid email.").trim(),
  password: z
    .string()
    .min(8, "At least 8 characters")
    .trim()
    .regex(/[A-Z]/, "At least one Uppercase letter")
    .regex(/[a-z]/, "At least one Lowercase letter")
    .regex(/\d/, "At least one Digit")
    .regex(/[\W_]/, "At least one Special character"),
});
```

## Database Models (Mongoose)

The server-side employs Mongoose to define the schemas for data stored in MongoDB. These schemas enforce data types and required fields for persistent data.

### `MangaSchema`

The `MangaSchema` defines the structure for manga documents within the database.

```javascript
import mongoose from "mongoose";

const MangaSchema = new mongoose.Schema({
  title: { type: String, required: [true, "please provide manga title"] },
  mangaId: {
    type: String,
    required: [true, "please provide manga id"],
    unique: true,
  },
  authorId: { type: String, required: [true, "please provide author id"] },
  description: { type: String },
});

export default mongoose.model("Manga", MangaSchema);
```

## Data Flow and Validation

The application follows a pattern where client-side schemas validate user input before transmission, and server-side models ensure data integrity upon storage.





```mermaid

    A["User Input"] --> B{Client-side Validation (Zod)};
    B -- "Passes Validation" --> C["API Request"];
    B -- "Fails Validation" --> D["Display Error Message"];
    C --> E{Server-side Processing};
    E --> F["Database Operation (Mongoose)"];
    F -- "Data Stored/Retrieved" --> G["API Response"];
    G --> H["Client-side Display"];
```



## Key Takeaways

*   **Client-side validation** with Zod provides immediate feedback to users and reduces unnecessary server load.
*   **Server-side Mongoose models** guarantee data consistency and integrity within the database.
*   Defined **form configurations** streamline the development of user interfaces and manage input behaviors.
*   **Clear schema definitions** are crucial for maintaining predictable data structures across the application.
