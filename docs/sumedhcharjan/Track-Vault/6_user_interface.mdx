# User Interface & Design

Track Vault utilizes a modern, modular frontend architecture built with Next.js, Tailwind CSS, and a composition-based component strategy. The design focuses on a "glassmorphism" aesthetic, combining backdrop blurs with a strict semantic color system to ensure accessibility and visual consistency across light and dark modes.

## Frontend Architecture

The application follows a hierarchical layout pattern, ensuring that global state providers and shared UI elements are persisted across page navigations.

```mermaid

    A["RootLayout"] --> B["Providers"]
    B --> C["Toaster"]

    B --> F["Footer"]
    D --> G["Auth State Logic"]
    E --> H["Page Components"]

### Root Layout
The `RootLayout` serves as the entry point for the application's DOM structure. It implements several critical global concerns:
- **Typography**: Integrates the `Inter` font for optimal readability.
- **Context Providers**: Wraps the application in a `Providers` component to manage global state.
- **Global Notifications**: Integrates `sonner` for non-blocking toast notifications.
- **Semantic Structure**: Uses a flex-column layout to ensure the `Footer` remains pushed to the bottom regardless of content length.

## Navigation & Authentication

The `Navbar` is implemented as a Server Component to handle authentication checks efficiently on the server side using Kinde Auth.

### Dynamic Rendering Logic
The navigation bar adapts its interface based on the user's authentication status:

| State | Visible Elements | Primary Action |
| :--- | :--- | :--- |
| **Unauthenticated** | About Link, Login Button | Redirect to Login |
| **Authenticated** | Your Files Link, User Avatar, Logout Button | Redirect to Dashboard |

The Navbar utilizes a floating design (`fixed top-0`) with `backdrop-blur-lg` to maintain context of the underlying content while providing a consistent navigation anchor.

## Reusable UI Components

Track Vault employs a compositional design pattern for its UI library, allowing developers to assemble complex interfaces from small, single-responsibility components.

### The Card System
The `Card` component is a prime example of this modularity. Rather than using a single monolithic component with numerous props, it is broken into sub-components:

- `<Card>`: The main container with border and shadow.
- `<CardHeader>`: Layout wrapper for the top section.
- `<CardTitle>` & `<CardDescription>`: Semantic text elements for hierarchy.
- `<CardAction>`: A positioned slot for buttons or menus within the header.
- `<CardContent>`: The primary body area.
- `<CardFooter>`: The bottom utility area.

## Design System & Theming

The application uses an advanced CSS variable system powered by Tailwind CSS and **OKLCH** color spaces, which provide more uniform perceived brightness across different hues.

### Theming Strategy
Colors are defined as CSS variables in `globals.css`, allowing for seamless switching between light and dark modes without rewriting class names.

- **Semantic Tokens**: Instead of hardcoded colors, the app uses tokens like `--color-primary`, `--color-muted`, and `--color-destructive`.
- **Dynamic Radius**: Corner rounding is standardized via a variable system (`--radius-sm` to `--radius-xl`), ensuring a consistent "soft" feel across all UI elements.
- **Base Layer**: A global CSS layer applies the `--border` and `--ring` colors to all elements by default to maintain a cohesive look.

### Styling Utility
The project uses a `cn` (class name) utility function to handle conditional Tailwind classes, preventing style conflicts and ensuring clean DOM output.
