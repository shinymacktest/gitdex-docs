# System Architecture

Doodle-Sync employs a microservices architecture designed for scalability, resilience, and maintainability. The system is composed of several independent services that communicate with each other through well-defined APIs and asynchronous messaging.

## Microservices Overview

The core services include:

*   **Discovery Server**: A central registry (Eureka server) that allows services to discover and communicate with each other.
*   **API Gateway**: Acts as the single entry point for all client requests. It handles routing, rate limiting, and request aggregation.
*   **User Service**: Manages user authentication, registration, and profile information.
*   **Game Service**: Orchestrates game logic, including room creation, player management, and game state transitions.
*   **Drawing Service**: Handles the real-time drawing canvas, synchronizing strokes across connected clients.
*   **Chat Service**: Facilitates in-game chat functionalities.
*   **Score Service**: Manages player scores and game results.
*   **Word Service**: Provides the words to be guessed during gameplay.

These services leverage Spring Boot and Spring Cloud for rapid development and integration.

## Deployment Strategy

The entire application is containerized using Docker and orchestrated with Docker Compose. This allows for consistent deployment across different environments and simplifies the management of dependencies.

The `docker-compose.yml` file defines the services, their configurations, networks, and dependencies. Key infrastructure components like PostgreSQL, MongoDB, Redis, and Kafka are also managed as containers within this setup.

```yaml
services:

  postgres:
    image: postgres:16-alpine
    container_name: scribble-postgres
    # ... other configurations

  mongodb:
    image: mongo:7
    container_name: scribble-mongodb
    # ... other configurations

  redis:
    image: redis:7-alpine
    container_name: scribble-redis
    # ... other configurations

  kafka:
    image: apache/kafka:latest
    container_name: scribble-kafka
    # ... other configurations

  discovery-server:
    build: ./discovery-server
    ports: ["8761:8761"]
    environment:
      SPRING_PROFILES_ACTIVE: docker
    # ... other configurations

  api-gateway:
    build: ./api-gateway
    container_name: scribble-gateway
    ports: ["8765:8765"]
    environment:
      SPRING_PROFILES_ACTIVE: docker
      SPRING_DATA_REDIS_HOST: scribble-redis
      JWT_SECRET: ${JWT_SECRET:-dev-secret-key-change-in-production-256bit}
    depends_on:
      discovery-server:
        condition: service_healthy
      redis:
        condition: service_healthy
    # ... other configurations
```

Each microservice is configured to connect to the Discovery Server (Eureka) for service registration and discovery. The API Gateway further utilizes this discovery mechanism to route requests to the appropriate downstream services.

## Interconnections and Communication

Communication between services primarily occurs through:

1.  **RESTful APIs**: The API Gateway exposes HTTP endpoints for client interactions, which are then forwarded to the respective microservices. Services can also communicate directly via REST if necessary.
2.  **WebSockets**: For real-time features like drawing synchronization and chat, WebSockets are utilized, with the Drawing Service and Chat Service exposing WebSocket endpoints.
3.  **Message Queues (Kafka)**: Asynchronous communication for events and data propagation between services is handled by Kafka. This decouples services and improves resilience.

The API Gateway configuration in `api-gateway/src/main/resources/application.yml` demonstrates how different paths are routed to specific services:

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: chat-service-route
          uri: lb://chat-service
          predicates:
            - Path=/chat/**

        - id: chat-service-ws
          uri: lb://chat-service
          predicates:
            - Path=/ws-chat/**

        - id: user-service-route
          uri: lb://user-service
          predicates:
            - Path=/user/**

        - id: game-service-route
          uri: lb://game-service
          predicates:
            - Path=/game/**

        - id: drawing-service-route
          uri: lb://drawing-service
          predicates:
            - Path=/draw/**

        - id: drawing-service-ws
          uri: lb://drawing-service
          predicates:
            - Path=/ws/**
```

The Discovery Server, implemented using Spring Cloud Netflix Eureka, is essential for this service discovery:

```java
package com.pm.discoveryserver;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.cloud.netflix.eureka.server.EnableEurekaServer;

@SpringBootApplication
@EnableEurekaServer
public class DiscoveryServerApplication {

    public static void main(String[] args) {
        SpringApplication.run(DiscoveryServerApplication.class, args);
    }

}
```

## System Flow (Example: User Login)

1.  A user sends a login request to the API Gateway.
2.  The API Gateway, configured to route `/user/**` requests to the `user-service`, forwards the request.
3.  The User Service validates the credentials and, if successful, generates a JWT.
4.  The User Service returns the JWT to the API Gateway.
5.  The API Gateway returns the JWT to the client.
6.  Subsequent requests from the client will include the JWT for authentication.

## Key Takeaways

*   **Decoupled Services**: Each microservice operates independently, allowing for easier development, deployment, and scaling.
*   **Centralized Discovery**: Eureka provides a robust mechanism for services to find each other.
*   **API Gateway as Entry Point**: Simplifies client interactions and enforces cross-cutting concerns like rate limiting.
*   **Containerization**: Docker and Docker Compose ensure consistent environments and simplified orchestration.
*   **Resilience**: The architecture is designed to be resilient, with fault tolerance mechanisms and asynchronous communication patterns.
